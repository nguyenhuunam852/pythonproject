<p>My take on this issue, i think is the nicest. I keep a my_filters.py on the templatetags directory. </p>

<pre><code>@register.filter(name='times') 
def times(number):
    return range(number)
</code></pre>

<p>And you would use like this:</p>

<pre><code>{% load my_filters %}
{% for i in 15|times %}
    &lt;li&gt;Item&lt;/li&gt;
{% endfor %}
</code></pre>

<p>You can pass a binding of</p>

<pre><code>{'n' : range(n) }
</code></pre>

<p>to the template, then do</p>

<pre><code>{% for i in n %}
...
{% endfor %}
</code></pre>

<p>Note that you'll get 0-based behavior (0, 1, ... n-1).</p>

<div class="answercell post-layout--right">

<p>Just incase anyone else comes across this question… I've created a template tag which lets you create a <code>range(...)</code>: <a href="http://www.djangosnippets.org/snippets/1926/" rel="noreferrer">http://www.djangosnippets.org/snippets/1926/</a></p>

<pre>
Accepts the same arguments as the 'range' builtin and creates a list containing
the result of 'range'.

Syntax:
    {% mkrange [start,] stop[, step] as context_name %}

For example:
    {% mkrange 5 10 2 as some_range %}
    {% for i in some_range %}
      {{ i }}: Something I want to repeat\n
    {% endfor %}

Produces:
    5: Something I want to repeat 
    7: Something I want to repeat 
    9: Something I want to repeat

</pre>

<div class="answercell post-layout--right">
    
    <div class="s-prose js-post-body" itemprop="text">
<p>This method supports all the functionality of the standard <code>range([start,] stop[, step])</code> function</p>

<p><code>&lt;app&gt;/templatetags/range.py</code></p>

<pre><code>from django import template

register = template.Library()


@register.filter(name='range')
def _range(_min, args=None):
    _max, _step = None, None
    if args:
        if not isinstance(args, int):
            _max, _step = map(int, args.split(','))
        else:
            _max = args
    args = filter(None, (_min, _max, _step))
    return range(*args)
</code></pre>

<p>Usage:</p>

<pre><code>{% load range %}

&lt;p&gt;stop 5
{% for value in 5|range %}
{{ value }}
{% endfor %}
&lt;/p&gt;

&lt;p&gt;start 5 stop 10
{% for value in 5|range:10 %}
{{ value }}
{% endfor %}
&lt;/p&gt;

&lt;p&gt;start 5 stop 10 step 2
{% for value in 5|range:"10,2" %}
{{ value }}
{% endfor %}
&lt;/p&gt;
</code></pre>

<p>Output</p>

<pre><code>&lt;p&gt;stop 5
0 1 2 3 4
&lt;/p&gt;

&lt;p&gt;start 5 stop 10
5 6 7 8 9
&lt;/p&gt;

&lt;p&gt;start 5 stop 10 step 2
5 7 9
&lt;/p&gt;
</code></pre>

<p>You should use "<a href="https://docs.djangoproject.com/en/dev/ref/templates/builtins/#slice" rel="noreferrer">slice</a>" in template, a example like this:</p>

<p>in views.py</p>

<pre><code>contexts = {
    'ALL_STORES': Store.objects.all(),
}

return render_to_response('store_list.html', contexts, RequestContext(request, processors=[custom_processor]))
</code></pre>

<p>in store_list.html:</p>

<pre><code>&lt;ul&gt;
{% for store in ALL_STORES|slice:":10" %}
    &lt;li class="store_item"&gt;{{ store.name }}&lt;/li&gt;
{% endfor %}
&lt;/ul&gt;
</code></pre>

<div class="answercell post-layout--right">
    
    <div class="s-prose js-post-body" itemprop="text">
<p>I'm just taking the popular answer a bit further and making it more robust.  This lets you specify any start point, so 0 or 1 for example.  It also uses python's range feature where the end is one less so it can be used directly with list lengths for example.</p>

<pre><code>@register.filter(name='range')
def filter_range(start, end):
  return range(start, end)
</code></pre>

<p>Then in your template just include the above template tag file and use the following:</p>

<pre><code>{% for c in 1|range:6 %}
{{ c }}
{% endfor %}
</code></pre>

<p>Now you can do 1-6 instead of just 0-6 or hard coding it.  Adding a step would require a template tag, this should cover more uses cases so it's a step forward.</p>

<div class="answercell post-layout--right">
    
    <div class="s-prose js-post-body" itemprop="text">
<p>This essentially requires a <code>range</code> function. A Django feature ticket was raised (<a href="https://code.djangoproject.com/ticket/13088" rel="noreferrer">https://code.djangoproject.com/ticket/13088</a>) for this but closed as "won't fix" with the following comment.</p>

<blockquote>
  <p>My impression of this idea is that it is trying to lead to programming in the template. If you have a list of options that need to be rendered, they should be computed in the view, not in the template. If that's as simple as a range of values, then so be it.</p>
</blockquote>

<p>They have a good point - Templates are supposed to be very simple representations of the view. You should create the limited required data in the view and pass to the template in the context.</p>

<p>This shows 1 to 20 numbers:</p>
<pre><code>{% for i in &quot;x&quot;|rjust:&quot;20&quot;|make_list %}
 {{ forloop.counter }}
{% endfor %}
</code></pre>
<p>also this can help you:
(count_all_slider_objects come from views)</p>
<pre><code>{% for i in &quot;x&quot;|rjust:count_all_slider_objects %}
  {{ forloop.counter }}
{% endfor %}
</code></pre>
<p>or</p>
<pre><code>  {% with counter=count_all_slider_objects %}
    {% if list_all_slider_objects %}
      {%  for slide in list_all_slider_objects %}
        {{forloop.counter|add:&quot;-1&quot;}}
        {% endfor%}
      {% endif %}
    {% endwith %}
</code></pre>
